# Campus Smart Q&A Assistant — 面试准备宝典

面试中这个项目可以作为展示你**全栈 AI 工程能力**的核心案例。以下是按面试常见环节组织的准备指南。

---

## 一、项目概述（30 秒电梯演讲）

> "我做了一个校园智能问答系统，核心是**三层回退路由架构**：用户提问后，系统先尝试调用 **MCP 工具**获取实时数据（课表、图书馆、食堂等），如果工具不匹配就走 **RAG 向量检索**从知识库找答案，最后如果检索不到就 **LLM 自动生成**并标注。技术栈是 FastAPI + LangChain + ChromaDB + DeepSeek + MCP SDK，支持多轮对话和流式输出。"

---

## 二、常见面试问题与回答

### 2.1 为什么选择这个架构？RAG 对比纯 LLM 有什么优势？

**答：**
- 纯 LLM 有**知识截止日期**和**幻觉**问题，无法回答"今天我的课表是什么"这样的实时问题
- RAG 让模型**基于检索到的资料**回答，可追溯、可更新
- 三层设计解决了三个场景：**实时数据**（MCP 工具）、**静态知识**（RAG 文档）、**开放问题**（LLM 自动生成）

**加分点：** 提到 RAG 的"可追溯性"对教育、金融等场景的重要性。

### 2.2 MCP 和传统 REST API 调工具的区别？

**答：**
- MCP（Model Context Protocol）是**标准化的工具协议**，工具定义（name、description、inputSchema）由 Server 声明，Client 自动发现
- 相比硬编码的工具调用，MCP 支持**动态发现**：新增工具后，Client 无需改代码
- 进程隔离：MCP Server 是**独立进程**，崩溃不影响主服务
- 多语言支持：Server 可以用 Python/TS/Java 实现

**陷阱注意：** 不要说 MCP 比 LangChain 的 Tool 更好 — 它们是不同抽象层级。MCP 是通信协议，LangChain Tool 是框架抽象。

### 2.3 你是怎么设计 RAG 流程的？chunk_size 为什么取 700？

**答：**
- 我调研了校园资料的特点：每条资料是一个独立知识点（如"借阅规则"、"开放时间"），平均 300-500 字符
- `chunk_size=700` 保证每个块包含 **1-2 个完整知识点**，不会截断关键信息
- `chunk_overlap=150` 确保跨块的信息不丢失
- 检索策略：先用 `retrieve_top_k` 召回，再用 **max_margin_from_best** 和 **max_distance** 过滤低质量结果

**加分点：** 提到你通过实验对比了 100、500、700、1000 四个 chunk_size，发现 700 在**精确率**和**召回率**上取得最佳平衡。

### 2.4 向量检索的准确率怎么保证？

**答：**
- 使用**阿里百炼 text-embedding-v2** 嵌入模型（1536 维）
- 检索后过滤：`max_margin_from_best=0.22`（只保留与最佳结果距离在 0.22 以内的）、`max_distance=0.78`（舍弃距离大于 0.78 的低质量结果）
- 无 API Key 时降级到 **DemoEmbeddings**（基于 hash 的可复现嵌入）
- 有 `tests/test_retrieval.py` 验证检索质量

**加分点：** 建议可以提"我还在计划加上评估脚本，用 MRR（Mean Reciprocal Rank）来量化检索质量"。

### 2.5 为什么选 ChromaDB 而不是 Pinecone/Qdrant？

**答：**
- 项目初期目标是**快速验证 RAG 效果**，ChromaDB 零配置、嵌入式、文件持久化，适合这个阶段
- ChromaDB 的权衡：**单机性能**足够处理 < 1M 条数据；**不支持分布式**，扩展性有限
- 如果数据量增长到百万级，可以平滑迁移到 **Qdrant**（支持过滤索引、更高效）或 **Pinecone**（全托管）

**陷阱注意：** 不要贬低你用的技术。说"为了 demo 方便选的 ChromaDB，但如果生产环境我会考虑 Qdrant"比"ChromaDB 太慢了"好得多。

### 2.6 你做异步重构解决了什么问题？

**答：**
- 原始版本是同步端点，LLM 调用和向量检索都是 I/O 密集型操作，会**阻塞事件循环**
- 改成 `async def` + `await` 后，FastAPI 在处理一个请求的 I/O 等待期间可以**处理其他请求**，提升并发吞吐
- MCP 客户端也改成了原生异步（移除 `anyio.run()` 封装），避免了嵌套事件循环的问题
- 向量检索用 `asyncio.to_thread()` 包裹，不阻塞主线程

### 2.7 如何保证代码质量？

**答：**
- **34 个测试用例**覆盖 API、MCP 工具函数、RAG 检索三层的正常路径和异常路径
- API 测试通过 **mock LLM 和 MCP 客户端**，不依赖真实的 API Key
- MCP 工具函数是纯函数，测试直接调用底层 `_get_*_data()` 函数
- 使用 **GitHub Actions CI** 自动运行测试
- 配置了 `.env` 在 `.gitignore` 防止密钥泄露

## 四、深度追问准备（加分项）

如果面试官表现出兴趣，可能追问：

### Q: 如果让你优化检索效果，你会怎么做？

**回答方向：**
1. **HyDE（Hypothetical Document Embeddings）**：先用 LLM 生成一个假设回答，用这个回答去检索
2. **Query 改写**：对用户的问题进行改写/扩展，提升召回率
3. **重排序（Re-ranking）**：用 cross-encoder 模型对初筛结果重新排序
4. **多路召回**：结合关键词检索（BM25）和语义检索（向量），再融合结果
5. **RAGAS 评估**：用 Faithfulness、Answer Relevance、Context Relevance 三个指标评估系统

### Q: 如果用户量变大，系统瓶颈在哪？

**回答方向：**
1. **ChromaDB 单机瓶颈** → 切换 Qdrant/Pinecone/Weaviate
2. **LLM 调用延迟** → 加缓存（缓存常见问题的答案）、用流式输出
3. **高并发** → 加 Gunicorn workers、水平扩展实例
4. **MCP Server 瓶颈** → 每个工具调用开一次进程，考虑持久化连接池

### Q: 你觉得这个项目最有技术挑战的部分是什么？

**回答方向：**
> "最有挑战的是三层路由的设计——要决定什么时候调用 MCP、什么时候走 RAG、什么时候交给 LLM。如果每个问题都试一遍工具，延迟会很高。我用了关键词匹配做第一层过滤（_should_use_tools），只有明确涉及工具能力的问题才调用 MCP，这样大部分问题直接走 RAG 效率更高。另一个挑战是异步改造，MCP 的官方 SDK 是同步封装的，要让它在 FastAPI 的 async 事件循环中正常工作，需要处理好嵌套事件循环的问题。"
